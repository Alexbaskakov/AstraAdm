# Теоретическая часть

**Контейнер** — это изолированное окружение для запуска приложения.  
Он содержит приложение и всё необходимое для его работы.

**Docker** — инструмент для создания и запуска контейнеров.

**Docker Swarm** — встроенный в Docker оркестратор, который:
- объединяет несколько серверов в кластер;
- распределяет контейнеры по узлам;
- автоматически перезапускает сервисы при отказе.

Основные понятия:
- **Node** — сервер в кластере (manager или worker).
- **Service** — описание того, как должен работать контейнер.
- **Stack** — набор сервисов, разворачиваемых вместе.
- **Replica** — количество копий сервиса.

# Практическая часть

## Исходные данные стенда
Используются Linux-хосты из предыдущих лабораторных работ:

- `Mgmt` — сервер управления (Manager)
- `Node1` — Linux-хост (Worker)
- `Node2` — Linux-хост (Worker)

Все узлы должны быть доступны по сети и иметь доступ в интернет.

---

## 1. Установка Docker

### 1.1 Установка Docker на всех узлах

Выполнить **на каждом узле** (`Mgmt`, `Node1`, `Node2`):

```
sudo apt update
sudo apt install docker.io -y
```

Проверка:

```
docker --version
```

### 1.2 Добавление пользователя в группу docker
```
sudo usermod -aG docker $USER
newgrp docker
```
Проверка:

```
docker ps
```
Если команда выполняется без sudo, всё настроено правильно.

# 2. Инициализация Docker Swarm
### 2.1 Инициализация кластера (только на Mgmt)
```
docker swarm init
```
В выводе будет строка вида:

```
docker swarm join --token SWMTKN-... 10.0.99.10:2377
```
⚠️ Скопировать эту команду, она нужна для подключения worker-нод.

### 2.2 Проверка состояния Swarm
```
docker node ls
```
Ожидаемо:

- 1 нода

роль: manager

статус: Ready

# 3. Добавление worker-нод в кластер
### 3.1 Подключение Node1 и Node2
Выполнить на Node1 и Node2 команду, полученную на шаге 2.1:

```
docker swarm join --token <TOKEN> <MANAGER_IP>:2377
```
### 3.2 Проверка кластера (на Mgmt)
```
docker node ls
```

Ожидаемо:

- 3 узла

- 1 manager

- 2 worker

- статус Ready

# 4. Подготовка Docker Stack
Для наглядности используется стек:

- nginx — веб-сервер

- replicas = 3 — для демонстрации отказоустойчивости

### 4.1 Создание файла docker-compose.yml
На сервере Mgmt:

```
mkdir ~/swarm
cd ~/swarm
```
```
cat > docker-compose.yml <<EOF
version: "3.8"

services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"
    deploy:
      replicas: 3
      restart_policy:
        condition: on-failure
    networks:
      - webnet

networks:
  webnet:
    driver: overlay
EOF
```
# 5. Развертывание стека
```
docker stack deploy -c docker-compose.yml webstack
```

Проверка:

```
docker stack ls
docker stack services webstack
docker stack ps webstack
```
# 6. Проверка работы сервиса
### 6.1 Проверка через браузер
Открыть в браузере:
```
http://IP_ЛЮБОГО_УЗЛА:8080
```
- Должна отображаться стандартная страница nginx.

### 6.2 Проверка через curl
```
curl http://localhost:8080
```
# 7. Распределение контейнеров по нодам
```
docker service ps webstack_web
```
Убедиться, что реплики распределены по разным узлам.

# 8. Моделирование отказа узла
### 8.1 Остановка Docker на одном worker-узле
На Node1:

```
sudo systemctl stop docker
```
### 8.2 Наблюдение за реакцией Swarm
На Mgmt:

```
docker node ls
docker service ps webstack_web
```
Ожидаемо:

- Node1 в состоянии Down

- контейнеры автоматически перезапущены на других узлах

### 8.3 Восстановление узла
На Node1:

```
sudo systemctl start docker
```
Проверка:

```
docker node ls
```
# 9. Удаление стека
После завершения лабораторной работы:

```
docker stack rm webstack
```
Проверка:

```
docker stack ls
docker service ls
```
# 10. Типичные ошибки
- Контейнеры не стартуют → проверь доступ в интернет.

- Порт не открывается → проверь firewall и правильность порта.

- Node в статусе Down → Docker не запущен.

- Stack не разворачивается → проверить синтаксис compose-файла.
