# Лабораторная работа №4
## Использование Jinja2 шаблонов для управления конфигурацией

## Цель лабораторной работы
Научиться генерировать конфигурационные файлы автоматически с помощью Jinja2-шаблонов и переменных Ansible, а затем применять эти конфиги на Linux-хостах через Playbook.

## Задачи лабораторной работы
1. Подготовить структуру проекта Ansible.
2. Создать переменные в `group_vars/` и (при необходимости) `host_vars/`.
3. Создать Jinja2-шаблоны для конфигов:
   - `/etc/chrony/chrony.conf` (NTP)
   - `/etc/ssh/sshd_config` (SSH)
4. Написать Playbook, который разворачивает шаблоны на хосты.
5. Запустить Playbook в режиме `--check` (dry-run).
6. Применить Playbook реально и проверить результат.
7. Сгенерировать текстовый конфиг MikroTik из шаблона (ознакомительно).

## Ожидаемый результат
После выполнения лабораторной работы студент умеет:
- хранить переменные отдельно от кода;
- писать шаблоны Jinja2;
- применять конфиги через модуль `template`;
- понимать, что изменится при запуске (`--check`);
- проверять, что конфигурация действительно обновилась.

# Теоретическая часть

**Jinja2** — шаблонизатор. В шаблоне можно писать обычный текст конфига, а переменные подставлять через `{{ ... }}`.  
Ansible берёт шаблон `.j2`, подставляет значения переменных и записывает готовый файл на удалённый хост.

**Почему так делают:**
- один шаблон на много серверов;
- меняем переменную → обновляется конфиг везде;
- меньше ручных ошибок.

# Практическая часть

### Что должно быть готово до начала
1. Есть сервер управления `Mgmt` (из ЛР1).
2. На нём установлен Ansible (из ЛР2).
3. Есть файл `hosts.ini`, в котором есть группа `[linux]` с вашими Linux-узлами.

Минимальный пример `hosts.ini` (проверь, что у тебя похоже):
```
[linux]
r1 ansible_host=10.0.12.1
r2 ansible_host=10.0.12.2
```

- Проверка связи:

```
ansible linux -i hosts.ini -m ping
```

- Если нет pong — дальше не идём, сначала чинить SSH и доступ.

# 1. Создание структуры проекта

Перейти в рабочий каталог Ansible и создать папки.

```
cd ~/ansible
```

# Создаём папки для шаблонов и переменных
```
mkdir -p templates group_vars host_vars
```

Проверка, что папки созданы:

```
ls -la
```

Должны быть **templates/**, **group_vars/**, **host_vars/**.

# 2. Создание переменных для группы Linux

Переменные группы linux кладём в group_vars/linux.yml.

```
cat > group_vars/linux.yml <<EOF
```

### Серверы времени для NTP

ntp_servers:
  - 0.pool.ntp.org
  - 1.pool.ntp.org

### Порт SSH (обычно 22)

ssh_port: 22
EOF

- Проверка содержимого:

```
cat group_vars/linux.yml
```

# 3. Создание шаблона для NTP
### 3.1 Создать шаблон
```
cat > templates/chrony.conf.j2 <<EOF
```

### Generated by Ansible. Do not edit manually.
```
{% for server in ntp_servers %}
server {{ server }} iburst
{% endfor %}

driftfile /var/lib/chrony/drift
makestep 1.0 3
rtcsync
EOF
```

### 3.2 Проверить, что файл на месте
```
ls -la templates/
cat templates/chrony.conf.j2
```

# 4. Создание шаблона для sshd_config

**Важно**: Если вы включите PasswordAuthentication no, то вход по паролю отключится.
Чтобы не заблокировать студентов, это включаем только если SSH-ключи уже настроены.

### 4.1 Создать шаблон
```
cat > templates/sshd_config.j2 <<EOF
```

# Generated by Ansible. Do not edit manually.

```
Port {{ ssh_port }}
Protocol 2
PermitRootLogin no
PasswordAuthentication no
UsePAM yes
EOF
```
### 4.2 Проверить

```
cat templates/sshd_config.j2
```

# 5. Создание Playbook для применения шаблонов

Playbook будет:

- копировать chrony.conf

- копировать sshd_config

- если файл изменился, перезапускать сервис

### 5.1 Создать Playbook
```
cat > linux_templates.yml <<EOF
---
- name: Apply configuration templates on Linux hosts
  hosts: linux
  become: true

  tasks:
    - name: Deploy chrony configuration
      template:
        src: templates/chrony.conf.j2
        dest: /etc/chrony/chrony.conf
      notify: restart chrony

    - name: Ensure chrony is installed
      apt:
        name: chrony
        state: present
        update_cache: true

    - name: Deploy SSH configuration
      template:
        src: templates/sshd_config.j2
        dest: /etc/ssh/sshd_config
      notify: restart ssh

  handlers:
    - name: restart chrony
      service:
        name: chrony
        state: restarted

    - name: restart ssh
      service:
        name: ssh
        state: restarted
EOF
```

### 5.2 Проверить

```
cat linux_templates.yml
```

# 6. Dry-run

Это самый безопасный шаг. Он показывает, что Ansible ХОЧЕТ изменить.

```
ansible-playbook -i hosts.ini linux_templates.yml --check
```

Ожидаемо:

- Playbook покажет changed на задачах template

- НО реально ничего не применит

- Если есть ошибки типа template not found — значит шаблон не там или имя неправильное.

# 7. Реальное применение Playbook
Если --check прошёл без ошибок, выполняем реальный запуск.

```
ansible-playbook -i hosts.ini linux_templates.yml
```

# 8. Проверка результата на Linux-узлах
Проверим, что файл реально изменился на удалённых узлах.

### 8.1 Проверка chrony.conf
```
ansible linux -i hosts.ini -m command -a "head -n 20 /etc/chrony/chrony.conf" --become
```
### 8.2 Проверка sshd_config
```
ansible linux -i hosts.ini -m command -a "head -n 20 /etc/ssh/sshd_config" --become
```
### 8.3 Проверка, что службы живые
```
ansible linux -i hosts.ini -m command -a "systemctl status chrony --no-pager" --become
ansible linux -i hosts.ini -m command -a "systemctl status ssh --no-pager" --become
```
Если active (running) — всё ок.

# 9. host_vars
Если нужно разные параметры для разных хостов, используется host_vars/<hostname>.yml.

Пример: сделать на r2 ssh_port = 2222.

```
cat > host_vars/r2.yml <<EOF
ssh_port: 2222
EOF
```
Dry-run:

```
ansible-playbook -i hosts.ini linux_templates.yml --check
```
Внимание: если вы реально поменяете порт, подключение по SSH может сломаться.
Это делается только осознанно. В учебной работе можно оставить 22.

# 10. MikroTik из шаблона
### 10.1 Шаблон конфигурации .rsc
```
cat > templates/mikrotik_config.rsc.j2 <<EOF
/system identity set name={{ inventory_hostname }}
/user add name=admin group=full password={{ mt_admin_password }}
EOF
```
### 10.2 Переменные MikroTik
```
cat > group_vars/mikrotik.yml <<EOF
mt_admin_password: StrongPassword
EOF
```
### 10.3 Генерация файла локально
```
ansible localhost -m template -a "src=templates/mikrotik_config.rsc.j2 dest=./mikrotik_{{ inventory_hostname }}.rsc" -e "inventory_hostname=LAB-MT" -e "@group_vars/mikrotik.yml"
```
Применение на RouterOS выполняется вручную командой:

```
/import file-name=имя_файла.rsc
```
